# 小型 OCR/上下文模型评估

## 结论

GLM-OCR 0.9B 和 HunyuanOCR 系列都是面向通用文档、文字、表格与版面理解的视觉语言模型，不是为五线谱、TAB、时值或吉他技法训练的 OMR 模型。它们不适合替代当前的谱表几何、事件定位和符号 CNN：体量远大于当前 59 万参数 TAB 检测器或约 88 万参数事件 CNN，而且没有直接输出弦号、品格、时值和技法关系的先验。

当前默认路径因此不下载或微调这两个大模型。它们可作为可选工具用于：

- OCR 标题、作曲者、速度文字、段落名和演奏说明；
- 在以后加入其他软件/扫描件时，作为离线教师产生版面或文字伪标签；
- 对低置信度页面给出辅助提示，但不能直接作为 GP 语义真值。

官方项目：

- [GLM-OCR](https://github.com/zai-org/GLM-OCR)
- [HunyuanOCR](https://github.com/Tencent-Hunyuan/HunyuanOCR)

## 乐谱上下文真正需要什么

视觉网络先输出候选事件，后续上下文只需要处理很短的离散序列，例如：

```text
measure, time_signature, onset, duration, dot, tuplet,
string, fret, tie, technique, confidence
```

这个任务更适合 4--6 层、10M--30M 参数的事件 Transformer，或直接使用小节时值动态规划。训练目标包括：

1. 小节时值闭合；
2. 同一事件的多弦音符合并；
3. 延音前后音高一致；
4. 多位品格数字和相邻指法连续性；
5. 推弦、滑音、击勾弦等技法的起止关系；
6. 对视觉低置信度候选重排，而不是凭空生成事件。

现阶段节奏 CNN 加确定性小节约束在指定目标曲的第 10 轮回归中达到 701/701 逐事件节奏正确；独立测试的主声部完整语义为 99.596%。因此再引入约 1B 级模型不会是性价比最高的默认方案。先补齐稀有技法、跨事件关系和跨软件数据；数据足够后，再用较大的模型作教师蒸馏事件 Transformer。目标曲参与了 hard-case 修正，其 100% 不能作为独立泛化成绩。

## 采用条件

只有满足以下条件才把上下文模型加入默认运行时：

- 独立来源测试集上的整事件准确率显著提高；
- 不降低罕见时值、休止和多声部召回；
- 所有改动都保留候选概率与修正审计记录；
- CPU 推理时间和模型体积仍符合本项目的轻量目标。
